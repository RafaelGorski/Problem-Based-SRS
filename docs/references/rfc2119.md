# RFC 2119 — Key Words for Use in RFCs to Indicate Requirement Levels

> **Reference collected from:** <https://www.rfc-editor.org/rfc/rfc2119>  
> **Collected (UTC):** 2026-04-29T14:37:00Z  
> **Updated by:** [RFC 8174](./rfc8174.md) (see BCP 14)

## Summary

RFC 2119 (BCP 14), authored by Scott Bradner (Harvard University, March 1997), defines key words used in standards-track documents to indicate requirement levels. These words, when written in **ALL CAPITALS**, carry specific normative meanings.

## Defined Key Words

| Key Word | Meaning |
|----------|---------|
| **MUST** | Absolute requirement of the specification |
| **MUST NOT** | Absolute prohibition of the specification |
| **REQUIRED** | Same as MUST |
| **SHALL** | Same as MUST |
| **SHALL NOT** | Same as MUST NOT |
| **SHOULD** | Valid reasons may exist to ignore, but implications must be understood and weighed |
| **SHOULD NOT** | Valid reasons may exist when behavior is acceptable, but implications should be understood |
| **RECOMMENDED** | Same as SHOULD |
| **NOT RECOMMENDED** | Same as SHOULD NOT |
| **MAY** | Item is truly optional |
| **OPTIONAL** | Same as MAY |

## BCP 14 Boilerplate (RFC 2119 + RFC 8174)

Documents using these key words SHOULD include this statement near the beginning:

> The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in BCP 14 [RFC 2119] [RFC 8174] when, and only when, they appear in all capitals, as shown here.

## Guidance on Use

Per RFC 2119 §6, these imperatives:
- MUST be used with care and sparingly
- MUST only be used where actually required for interoperation or to limit harmful behavior
- MUST NOT be used to impose a particular method where not required for interoperability
