üìä The 5-Step Methodology

A systematic process that ensures you discover problems first, then design solutions. Each step builds on the previous to maximize your impact:

Step 1: Customer Problems (CP)

Identify the WHY - understand business pain and classify by problem class (Obligation / Expectation / Hope ‚Äî roughly must-have / should-have / could-have)

‚úÖ You get: Clear focus on what matters most, justification for every feature

/cp

Step 2: Software Glance

Sketch high-level solution approach - components, interfaces, boundaries

‚úÖ You get: Shared understanding with team, architectural direction

/glance

Step 3: Customer Needs (CN)

Define the WHAT - outcomes the system must deliver to solve the problems

‚úÖ You get: Success criteria, measurable goals, scope boundaries

/cn /zigzag (validate)

Step 4: Software Vision

Detail architecture, features, constraints, and technical approach

‚úÖ You get: Technical roadmap, stakeholder alignment, design decisions

/vision

Step 5: Functional Requirements (FR)

Define the HOW - detailed, testable requirements for implementation

‚úÖ You get: Implementation checklist, acceptance criteria, full traceability

/fr /zigzag (validate)
üí° Why This Order Matters:
Starting with problems (WHY) before solutions (HOW) ensures you build the right thing. Engineers who skip problem discovery often build features that look good but don't solve actual needs.

üìä Problem Classification

Customer Problems are classified by severity to prioritize what must be solved:

  • Obligation (Must/Required): High priority - severe consequences if unsolved (legal/contractual)
  • Expectation (Should/Expects): Medium priority - moderate impact if unsolved (business goals)
  • Hope (Wishes): Low priority - minimal penalty if unsolved (improvements)

How engineers use Problem-Based SRS in different situations:

üÜï New Feature Development

üí° Problem Discovery

üîß Integration with Development Workflow

During Sprint Planning

  • Use /cp to capture user stories as customer problems
  • Use /cn to define acceptance criteria
  • Use /fr to break down into technical requirements

During Feature Design

  • Reference /glance for high-level approach
  • Reference /vision for architectural decisions
  • Use /zigzag to ensure design addresses all needs

During Implementation

  • Keep /fr requirements as implementation checklist
  • Trace each requirement back to CN and CP for context
  • Update requirements if new problems discovered

During Code Review

  • Verify implementation satisfies /fr requirements
  • Check requirements still trace to customer needs
  • Validate no scope creep beyond identified problems
